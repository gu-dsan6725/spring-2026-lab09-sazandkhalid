For this trace, I connected to an MCP server providing documentation tools (resolve-library-id and query-docs). In Braintrust, I could see the full parent-child hierarchy: the root invoke_agent span contained three execute_event_loop_cycle children, each holding a chat LLM call and in two cases a tool call. Compared to the DuckDuckGo traces, MCP tool calls required two sequential tool invocations before the final answer: the agent first had to resolve the library identifier, then use that result to query the actual documentation. This chaining meant an extra loop cycle, pushing total tokens to 19,013 (vs ~4,328 for DuckDuckGo) and total latency to 7.74s (vs 5.93s). The tool calls themselves were fast (~1.08s and ~1.36s), so the bottleneck was still the LLM calls — especially the final chat at 1.8s with the largest context of 4,939 tokens.
